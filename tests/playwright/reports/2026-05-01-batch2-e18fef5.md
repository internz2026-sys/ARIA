## Diagnosis (failures only)

**Case 1 — bulk-cancel still hard-deletes despite Coder A's fix.**

The frontend modal copy is now correct ("Move 2 items to Cancelled?", purple `#534AB7` button, "restored from the Cancelled tab" body). But clicking confirm fires `DELETE /api/inbox/{id}` per item — observed via network log — and the rows ARE permanently removed:

- All count: 128 → 122 across two test runs (4 items vanished)
- Cancelled tab: shows "No deliverables yet" both immediately and on full reload
- `GET /api/inbox/{tenant}/counts` response has no `cancelled` key

This contradicts Coder A's diagnosis from the previous batch ("backend already soft-cancels via DELETE without `?permanent=true`"). Either:
1. The backend DELETE handler's `permanent: bool = False` default isn't actually soft-cancelling for some reason (maybe the CORS preflight is rewriting query params? Maybe the row never had a cancelable status? Maybe a service-side handler is intercepting?), OR
2. The DELETE handler's "soft-cancel" path silently degrades to hard delete on error and is failing silently, OR
3. The bulk path is hitting a different endpoint than single-row delete — both ostensibly call `inbox.remove(id)`, but worth verifying with the network log.

**Recommended next step:** spawn an Opus backend agent to read `backend/routers/inbox.py` DELETE handler line-by-line, run the actual SQL the soft-cancel branch executes, and verify the row's status in the DB right after the DELETE call. Single-row delete also needs re-verification — the Playwright tester only re-checked the bulk path this round.
