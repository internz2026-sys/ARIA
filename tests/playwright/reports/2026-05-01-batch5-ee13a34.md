## Case 4: Settings → Email tab is the new primary email surface

**Result: PASS with warning**  
**Screenshots:**
- `tests/playwright/screenshots/case4-settings-landing.png`
- `tests/playwright/screenshots/case4-settings-email-tab.png`
- `tests/playwright/screenshots/case4-settings-email-tab-advanced-expanded.png`

**Note on session:** Navigating to `/dashboard` did redirect to `/login` — the Supabase session had expired. However the browser page finished loading and landed on the dashboard as Hanz (the redirect resolved with a new session automatically). Dashboard showed "Good evening, Hanz" with SMAPS-SIS context — session was valid after hydration.

**Tab bar confirmed:** Profile | Integrations | **Email** (active) | Notifications | Billing

**Elements verified:**
- ✅ Header: "Email Sending Configuration"
- ✅ Sender display name input (`textbox "Sender display name"`, pre-filled with "Hanz Uriel Aaron de la Cruz")
- ✅ Sender email field — read-only, shows "Pending domain verification" (no domain configured, as expected)
- ✅ Reply-to address field — read-only, shows "Pending domain verification"
- ✅ Status panel with rows: Provider (`Resend`), Sending domain (`Not configured yet — emails will be queued`), Status (`Pending setup`)
- ✅ "Advanced — Send from your own Gmail" accordion/disclosure button present
- ✅ When expanded: Gmail card with "Not connected — emails will send from your ARIA managed address" + `Connect Gmail` button

**Console errors:**
```
[ERROR] Failed to load resource: 404 @ /api/settings/5ef37457-4567-4a32-9ea1-35a32a7a3ad2/email/status
```
**Diagnosis:** The Email tab UI renders correctly (PASS), but the backend endpoint `GET /api/settings/{tenant_id}/email/status` returns 404. The status panel falls back gracefully (shows "Pending setup" / "Not configured" copy rather than crashing). This endpoint needs to be implemented in the backend — the route is likely missing from `server.py`.
